# ADR-0020: X1.2 — Gmail-inbox → GRN auto-match bridge

- Status: **Accepted**
- Date: 2026-04-17
- Deciders: Sourav Shaw (founder), future tech lead
- Playbook ref: v2.0 §4 (X1 moat), §8.1 (locked stack)
- Supersedes: nothing
- Builds on: ADR 0001 (X1 Gmail-to-GRN), A1-A6 GRN schema

## Context

X1.1 shipped: owner connects Gmail, we fetch attachment text, apply a
`SupplierTemplate` regex → we have `ParsedBill.lines[]` with
`{ productHint, batchNo, expiryDate, qty, ratePaise, mrpPaise, gstRate,
confidence }`. The handoff bus (`setPendingGrnDraft` → `takePendingGrnDraft`)
delivers the parsed lines to `GrnScreen` on the next mount.

**Current gap:** `GrnScreen` shows parsed lines in a read-only banner.
The owner still has to search and re-add each product manually — the exact
friction X1 was supposed to eliminate. We converted "type the whole bill"
into "read-and-retype the whole bill."

## Decision

Add a pure-TS bridge package `@pharmacare/gmail-grn-bridge` that converts
`ParsedLine + ProductHit[]` → `LineMatch` (matched / unmatched + confidence)
with deterministic matching rules. Then `GrnScreen` auto-populates its
`DraftLine[]` with matched products; unmatched lines stay visible in the
import banner with per-line "Skip" (and future "Create product") actions.

### Matching rules (locked)

Applied in order; first match wins. All comparisons normalise the hint:
lowercase, strip non-alphanumerics except spaces, collapse whitespace.

1. **exact-name** (conf = 0.95): normalised hint equals normalised
   `product.name` of some candidate. Unambiguous.
2. **token-overlap** (conf = Jaccard × 0.85, floor 0): Jaccard similarity
   on token sets ≥ `0.6`. Best candidate wins; ties broken by highest
   Jaccard, then shortest name (prefer more specific match).
3. **hsn-assist** (conf = Jaccard × 0.6, floor 0): if any candidate's HSN
   matches the parsed HSN (when present) AND token-Jaccard ≥ `0.3`, keep
   as a weaker match. Distinguishes same-molecule strengths/packs by HSN.
4. **none** (conf = 0, kind = unmatched): nothing above threshold.

Confidence chip thresholds (UI):
- `≥ 0.80` — green "high" chip
- `0.50 – 0.79` — amber "medium" chip; user should double-check
- `< 0.50` — red "low" chip; shown but requires user confirmation before
  save (disables row in `canSave` gate unless user acknowledges)

### Auto-populate policy

`GrnScreen` on import-mount:

1. For each parsed line, call `searchProductsRpc(hint, 5)` and feed the
   returned `ProductHit[]` to `matchParsedLine(hint, hsn, candidates)`.
2. If `kind === "matched"`, synthesise a `DraftLine` with
   `productId = match.product.id`, `qty`, `batchNo`, `mfgDate ?? ""`,
   `expiryDate`, `purchasePricePaise = ratePaise`,
   `mrpPaise = line.mrpPaise ?? product.mrpPaise`. Append to `lines`.
3. Keep `unmatchedLines[]` separately in the banner with inline
   `Skip` button (drops from banner) + `Create product` button (wired in
   X1.3 — today it's a stub with a tooltip).
4. On save, `canSave` still runs the existing validator per line; low-conf
   chips require user `Acknowledge` click to clear the gate.

### Non-goals (out of scope for X1.2)

- **Automatic product creation** from unmatched lines → X1.3.
- **ML-based matching** (LayoutLMv3/Donut) → Tier B, ADR 0001 phase 2.
- **Cross-line dedupe** (same SKU listed twice in one bill) → deferred.
- **Rust-side matching** — keeping the bridge pure-TS means we can unit
  test with zero IPC, and the match volume (≤ 100 lines/bill × 60 bills/wk)
  is trivially fast in-process.
- **New migrations / Tauri commands** — reuses existing
  `search_products` + handoff bus.

### Locked design choices

- **Bridge runs in TS, not Rust.** Tests use no Tauri; package has zero
  runtime deps; `shared-types` only.
- **`searchProductsRpc` stays unchanged.** No new FTS-scope flag needed —
  the bridge adapts the existing top-5 result to a `LineMatch`.
- **Bridge is stateless.** No caching, no learning. X1.3 adds a
  per-supplier "last-matched-SKU" hint table.
- **Unmatched lines are not silently dropped.** They remain in the banner;
  the owner must Skip or Create each one before the banner disappears.
- **No external HTTP** from the bridge. Everything happens on the local
  lookup surface.

## Consequences

**Positive**
- The core X1 moat promise — "Gmail → one-click GRN" — becomes real for
  lines the shop has seen before (Tier A high-volume suppliers, the 60%
  audience per ADR 0001).
- Pure-TS + pure-function matching is a 1-file unit-test story; no CI
  infra changes.
- Low-conf / unmatched path stays **user-gated**, honouring "AI always
  has a manual fallback" principle (§12).

**Negative / Risk**
- Two token-overlap buckets (medium/low) can silently mis-match when the
  same molecule has multiple strengths (Paracetamol 500 vs 650). The HSN
  tiebreak + the confidence chip is the mitigation; user must confirm
  low-conf rows.
- `searchProductsRpc(hint, 5)` is the bottleneck for recall. If a bill's
  hint is noisier than the FTS5 tokeniser expects, we miss. X1.3 adds a
  fallback "contains-any-bigram" scan over active products.
- Bridge doesn't know about MRP variance — if a parsed line's MRP
  disagrees with the matched product's MRP, we silently accept the
  parsed one. Future: flag `mrp-drift` chip.

## Alternatives considered

1. **Server-side match in Rust command `prepare_grn_from_email`** —
   would encapsulate fetch + parse + match in one IPC. Rejected: couples
   the match logic to Tauri, hurts unit-testability, and the parse path
   already lives in TS via `test_supplier_template`.
2. **Use `@pharmacare/sku-images` phash for visual match** — only useful
   once X2+X3 converge. Deferred to X1.3.
3. **Require user to explicitly trigger auto-match** — rejected; the
   whole point is the single-click flow. Low-conf rows still gate save.

## Rollout plan

- Phase 1 (this PR): bridge package + GrnScreen integration. Tier A
  suppliers only.
- Phase 2 (X1.3): "Create product" inline action; `mrp-drift` chip;
  per-supplier match-history table.
- Phase 3 (X1.4+): Tier B edge model wired through the same bridge
  interface.
