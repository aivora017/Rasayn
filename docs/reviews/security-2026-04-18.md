## Findings

### S01 [HIGH] OAuth loopback callback does not verify `state` parameter
**File**: `apps/desktop/src-tauri/src/oauth/mod.rs:76-90` generates `state_token`; `apps/desktop/src-tauri/src/oauth/loopback.rs:60-76` (`parse_code_from_request`) only extracts `code`.
**Risk**: RFC 6749 §10.12 CSRF — a hostile page/local process that can send a request to the ephemeral loopback port (during the ~5-minute window) could inject an attacker-controlled auth code, causing the pharmacy's desktop client to exchange and bind a Google account the owner never consented to. Installed-app + PKCE limits the blast radius (the attacker can't replay outside the session), and the window is narrow, but the callback accepts ANY `code=` request from any source on 127.0.0.1. On multi-user Windows machines (terminal-server pharmacies) this is reachable from other local users.
**Evidence**:
```rust
// mod.rs:76 — token generated
let state_token = format!("{:x}", /* nanos */);
let auth_url = google::build_auth_url(..., &state_token);
// loopback.rs:68 — only `code` is pulled
for kv in query.split('&') {
    if let Some((k, v)) = kv.split_once('=') {
        if k == "code" { return Some(url_decode(v)); }
    }
}
```
**Fix**: Thread `state_token` into `await_code`, parse `state=` alongside `code=`, reject with 400 if mismatch. Also tighten loopback to require `GET /callback` path (currently any path accepted).
**Effort**: S (~1 hr incl. test).

### S02 [HIGH] PNG/WebP decompression bomb via unbounded `image::load_from_memory`
**File**: `apps/desktop/src-tauri/src/phash.rs:26` (`image::load_from_memory(bytes)`); called from `images.rs:208` (`phash::compute_phash`) inside `attach_product_image`.
**Risk**: The 2 MiB input cap is enforced on the *encoded* bytes. PNG DEFLATE can easily hit compression ratios >1000:1 on zero-filled or repeating pixel data; a crafted 2 MiB PNG can request allocation of multi-GB pixel buffers, triggering OOM / process kill on the 4 GB Win7 target (Hard Rule 7). `image = 0.25` exposes `Limits` (`max_alloc`, `max_image_width`, `max_image_height`) but the call site uses the no-limits convenience API.
**Evidence**: `Cargo.toml:24` `image = { version = "0.25", ... }` — no `Limits` set anywhere in the crate. Attacker path: owner uploads image (intentional or via forwarded Gmail attachment future path) → phash compute panics/OOMs → desktop dies mid-transaction.
**Fix**: Replace `image::load_from_memory(bytes)` with a `Reader` + `limits(Limits::default())` or custom small limits (e.g., 8192×8192, 256 MiB alloc). Same fix applies to any future image decode path.
**Effort**: S (~30 min).

### S03 [MEDIUM] Gmail API responses have no byte-size cap on body
**File**: `apps/desktop/src-tauri/src/oauth/gmail_api.rs:48-53` (client builder), `fetch_attachment` at 195.
**Risk**: The only bound is a 30 s HTTP timeout. A hostile proxy, MitM, or a compromised Gmail account returning a multi-GB attachment JSON would buffer the entire `resp.json()` / `resp.text()` body in RAM. Gmail's own server-side 25 MB attachment cap is enforced only at the Gmail side — if we ever switch endpoints (e.g. `ProxyHTTPClient` for testing, corporate TLS-inspection appliance) there's nothing local. Not a direct SSRF (URL is hard-coded to `gmail.googleapis.com`) but a data-plane amplification.
**Evidence**: `fetch_attachment` does `resp.json::<AttResp>()` with no content-length preflight. `list_messages` similarly unbounded.
**Fix**: (a) Add `resp.content_length()` check ≤ 30 MB before `json()`; (b) optionally stream with `.bytes_stream()` + accumulator; (c) at minimum emit a `tracing::warn` for any attachment with `size > 10 MB` in the meta payload.
**Effort**: S.

### S04 [MEDIUM] OAuth audit payload is ad-hoc string concat → malformed JSON / log injection if email contains `"`
**File**: `apps/desktop/src-tauri/src/oauth/mod.rs:125`
**Risk**: `format!(r#"{{"email":"{}"}}"#, email)` writes attacker-influenced data (the email from the id_token, which we explicitly do NOT verify the JWT signature of — see `google.rs:163`) into `audit_log.payload` as unescaped JSON. A hostile id_token with an `email` field like `x","role":"admin` will corrupt downstream audit tooling that parses `payload` as JSON, potentially allowing a forged second field to be read. Contrast with the `gmail_list_messages` paths at line 228/269 which correctly use `serde_json::Value::String(...)`.
**Evidence**:
```rust
audit(&c, &shop_id, "gmail_connect",
      &format!(r#"{{"email":"{}"}}"#, email));
```
**Fix**: Use `serde_json::json!({"email": email}).to_string()` everywhere in `audit()` callers.
**Effort**: S.

### S05 [MEDIUM] `pendingGrnDraft` module singleton races on fast remount
**File**: `apps/desktop/src/lib/pendingGrnDraft.ts:18-26`; consumer `apps/desktop/src/components/GrnScreen.tsx:78-129`.
**Risk**: The draft is a module-level `let pending`. `take` is destructive. Under React.StrictMode (dev) or any remount pattern, two mounts race: (1) if both run `take`, one gets `null`; (2) if mount→unmount→mount happens after `setPendingGrnDraft` but before the first effect completes, the second mount may steal the draft mid-auto-match and the first mount continues with stale closure state. The `cancelled` flag inside the effect does not clear `pending`.
**Evidence**: `useEffect(() => { const d = takePendingGrnDraft(); ... })` — draft is gone the instant the first effect fires; any re-render-triggered second effect sees null. Also the match loop at GrnScreen.tsx:88-125 calls `searchProductsRpc` twice per line (once for match, once to re-find by id — see line 108), doubling IPC traffic.
**Fix**: (a) Keep the draft until explicit "dismiss" or "save"; move to `sessionStorage` or a small in-memory store keyed by sourceMessageId. (b) Dedupe the double `searchProductsRpc` call — capture the hit during the first call.
**Effort**: M.

### S06 [MEDIUM] `gmail_disconnect` does not await the blocking `revoke`, so refresh token may linger at Google if keyring delete succeeds but revoke fails silently
**File**: `apps/desktop/src-tauri/src/oauth/mod.rs:171-185`
**Risk**: `let _ = tokio::task::spawn_blocking(...).await;` discards the result. Keyring deletion proceeds regardless. If Google's revoke endpoint returns 500, the refresh token remains valid at Google (can be used if ex-filtrated from keyring backup) even though the local UI shows "disconnected". Compliance posture: we're telling the user/auditor the account is revoked when it might not be.
**Fix**: Surface revoke failure via tracing + audit row; optionally retry-once before keyring delete.
**Effort**: S.

### S07 [MEDIUM] `get_duplicate_suspects` O(N²) scales to 12.5M pairs in-memory; fine at pilot but unbounded result set
**File**: `apps/desktop/src-tauri/src/images.rs:485-533`
**Risk**: ADR-0019 acknowledges the O(N²) quadratic. That's fine. Separately, there is no cap on the number of result rows pushed into `out` — a catalog of truly-similar-packaging products (e.g., generic paracetamol strips from 50 vendors) could produce >5000 pairs, each a struct-with-two-strings, piped back to the UI which renders all rows in one table (ComplianceDashboard.tsx:140-161 has no pagination). UI becomes unresponsive.
**Fix**: Cap `out.len() <= N_MAX` (e.g. 200) and return a `truncated: bool` flag; add "…more hidden" row in the UI.
**Effort**: S.

### S08 [LOW] `products.rs:173-184` uses `format!` to inline a constant column list + constant `strftime()` SQL function — safe but looks like injection
**File**: `apps/desktop/src-tauri/src/products.rs:173-207, 219`
**Risk**: None in practice — `SELECT_COLS` is a compile-time string literal and `now` is the literal `strftime('%Y-%m-%dT%H:%M:%fZ','now')`. Flagged because `format!(... SELECT ...)` is the shape future reviewers will grep for and misread. Zero user data goes through `format!`; all parameters are `params![...]`.
**Fix**: Convert to `const SQL: &str = "... VALUES(... strftime(...) ..."` so there's no `format!` near SQL at all.
**Effort**: S.

### S09 [LOW] `loopback.rs::url_decode` casts bytes to chars (`out.push(b as char)`) — not UTF-8 safe
**File**: `apps/desktop/src-tauri/src/oauth/loopback.rs:78-101`
**Risk**: Google auth codes are ASCII so practically benign. If Google ever changes the shape, or the function is reused for other query params (e.g., `state` per S01), a multi-byte UTF-8 sequence will be mangled into Latin-1. Errors would surface at the token-exchange step as opaque `invalid_grant`.
**Fix**: Decode into `Vec<u8>`, then `String::from_utf8_lossy`.
**Effort**: S.

### S10 [LOW] `extract_email_from_id_token` does zero signature validation
**File**: `apps/desktop/src-tauri/src/oauth/google.rs:163-175`
**Risk**: Comment acknowledges "No signature check — we trust the token only to label the connected account for the UI." Given we trust Google's TLS endpoint returned the id_token, that's defensible. The remaining risk is future reuse — e.g., if someone copies this helper to validate an email for authorization purposes. Leave a more pointed warning on the function.
**Fix**: Rename to `extract_email_from_id_token_unverified` and add a `#[deprecated]` note if used for any auth decision.
**Effort**: S.

### S11 [LOW] `sanitize_filename` keeps trailing dots; attacker-controlled filename could be `"..."` → collapses to `"..."`
**File**: `apps/desktop/src-tauri/src/oauth/gmail_api.rs:250-265`
**Risk**: `sanitize_filename` preserves `.` and `-` and `_`. Test case `"../../etc/passwd"` → `.._.._etc_passwd"` correctly neuters slashes; however, a Windows filename `"CON.txt"` or trailing-space / trailing-dot abuse would still be allowed. Path is joined under `temp_dir().join("pharmacare-gmail")` so traversal is blocked, but reserved names could cause write failures that are currently logged as generic errors.
**Fix**: Reject Windows reserved base names and collapse leading/trailing dots; if input is only `.` chars, fall back to `"attachment.bin"`.
**Effort**: S.

### S12 [LOW] `url_decode` in `loopback.rs` mis-handles `%` at end of string
**File**: `apps/desktop/src-tauri/src/oauth/loopback.rs:82-100`
**Risk**: `if bytes[i] == b'%' && i + 2 < bytes.len()` — the `< bytes.len()` guard requires THREE remaining bytes (i, i+1, i+2 — so `i + 2 < len` means at least `i+3 <= len`, which is correct). But when the 2 hex digits are invalid (e.g. `%GG`), the code falls through and writes the raw `%` byte without consuming hex chars — not quite idempotent if someone tries to double-decode, and silently drops errors. Minor.
**Fix**: Return `Option<String>` and bubble malformed-percent-encoding as a 400 response.
**Effort**: S.

## Non-issues (things I checked that are fine)

* **SQL injection across the three PRs**: every `execute` / `query_row` uses `params![]` or positional parameters. The two `format!`-built SQL strings (`products.rs:173, 207, 219`) inline only compile-time constants; no user data. `oauth/mod.rs::audit` uses bound `?1/?2/?3` for actor/entity/payload.
* **Server-side MIME sniff**: `images.rs::sniff_mime` ignores client-reported MIME, enforces PNG/JPEG/WebP magic, matches TS mirror in `packages/sku-images/src/mime.ts`. Rejects GIF, empty, and short inputs (unit tests cover).
* **Server-side SHA256 re-hash**: client hash never persisted — `images.rs:165` recomputes. Confirmed.
* **Schedule H/H1/X gate cannot be skipped via image delete**: `delete_product_image` runs `DELETE FROM product_images` which triggers `trg_product_images_sync_after_delete` → sets `products.image_sha256=NULL` → `trg_products_schedule_img_upd` aborts the transaction with the "Schedule H/H1/X product requires image_sha256" error. Verified trigger chain in `0001_init.sql:66-72` + `0017_x2_product_images.sql:66-70`. Rust error translation path at `images.rs:334-341` surfaces a friendly message. BYPASS NOT POSSIBLE via this surface.
* **Schedule gate during attach**: `attach_product_image` runs inside a transaction; `INSERT INTO product_images` fires `trg_product_images_sync_after_insert` which updates `products.image_sha256` to the new hash. Both triggers compose atomically.
* **Integer overflow on image dims**: `width_px` / `height_px` columns exist in `0017` but are NEVER populated (images.rs never sets them). So no current overflow path. (Note: if you add dim extraction later, guard against `u32::MAX`.)
* **Audit append-only**: `trg_product_image_audit_no_update` + `trg_product_image_audit_no_delete` reject UPDATE/DELETE. Good.
* **Refresh token storage**: OS keyring via `keyring = "3"` (Windows: DPAPI-backed Credential Manager). Never written to SQLite, never logged. Verified `keyring_store.rs` is a thin wrapper.
* **Refresh flow**: `access_token_for` reads refresh from keyring, exchanges for a fresh access token per call, discards. Access tokens never persisted. Good pattern.
* **pHash panic-free on garbage**: `compute_phash` returns `Err(String)` on image-crate decode failure; caller in `attach_product_image:208-218` catches and stores `NULL`. `hamming_distance` returns `Err` on malformed input; both `find_similar_images` and `get_duplicate_suspects` `continue` on Err rather than failing the whole query. Confirmed soft-fail on golden-set tests.
* **pHash DC-term exclusion**: `phash.rs:58` slices `[1..]` correctly; median of remaining 63 values uses indices 31,32 of the sorted vec = the median of a 63-element sample (0-indexed index 31 is the median). This is correct.
* **`useEffect` cleanups**: GrnScreen.tsx:127 returns `() => { cancelled = true; }` for the async match loop. The toast effect (131-135) returns `clearTimeout`. F9 handler (215-221) removes its listener. ComplianceDashboard only runs async-fetch; no subscriptions to clean up.
* **Matcher is pure + total**: `matchParsedLine` never throws on empty hint, zero candidates, null HSN, non-ASCII; tests cover these.
* **Gmail query input encoding**: `gmail_api::urlencode` is RFC 3986-compliant per unit test; user query text safely encoded.

## Recommendations summary

Prioritised action list (before pilot install):

1. **S01 — Verify OAuth `state` on loopback callback.** Blocker-ish on multi-user Windows hosts. ~1 hr.
2. **S02 — Set `image::Limits` before `load_from_memory`.** Trivial fix; removes decompression-bomb DoS. ~30 min.
3. **S03 — Cap Gmail response body size (≤30 MB) + warn above 10 MB.** Defensive; 30 min.
4. **S04 — Switch OAuth audit payload to `serde_json::json!`.** Fixes JSON-injection via email. 15 min.
5. **S05 — `pendingGrnDraft` persistence + dedupe `searchProductsRpc`.** Correctness; cleaner race story. 1-2 hr.
6. **S06 — Surface `revoke` failure.** Compliance posture polish. 20 min.
7. **S07 — Cap `get_duplicate_suspects` result set + pagination hint.** UI resilience. 30 min.

Backlog (S08–S12): opportunistic polish; none compromise pilot safety.

If only three of these land before first outside-tenant install: **S01, S02, S04**. Together they close the only real CSRF vector, the only memory-DoS vector, and the only log-integrity vector in the three-PR surface.
